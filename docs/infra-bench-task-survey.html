Full instruction 
You are working on a high-performance FP8 GEMM kernel for an AMD MI350 GPU
(ROCm), written in Triton's **Gluon** dialect. The kernel computes

```
C[M, N] = A[M, K] @ dequant(W_fp8[N, K]).T
```

where `A` is bf16 activation and `W` is an FP8 (`e4m3`) weight that must be
dequantized with a scale before use.

The kernel at `/app/gemm_splitk.py` currently dequantizes the weight with
**block-wise** quantization. Your job is to change **only the FP8 weight
quantization method** to **per-token / PTPC** (per-output-channel) quantization.
Do not change the bf16 path, the tiling/MFMA structure, or the function name and
signature of `gemm_splitk_kernel`.

## The two quantization methods

**Block-wise (current):** the weight scale depends on BOTH the output block and
the K block. Scale tensor has shape `[N // 128, K // 128]`, and

```
W_ref[n, k] = W_fp8[n, k] * W_scale[n // 128, k // 128]
```

Because the scale changes along K, the current kernel reloads a new scale every
128 elements of K and multiplies it into the partial accumulation inside the
K loop.

**Per-token / PTPC (target):** the weight is quantized along its last dim (K),
so the scale is **per output channel** and does NOT depend on K. Scale tensor
has shape `[N, 1]` (equivalently `[N]`), and

```
W_ref[n, k] = W_fp8[n, k] * W_scale[n]
```

This follows the standard per-token quant relation (quantize along the last
dimension, one scale per row):

```
Y_quant = X_float / scale,   scale = row_amax / dtype_max
```

so for a weight `[N, K]`, reducing over K gives one scale per output row `n`.

## What to change in the kernel

Because the PTPC weight scale is independent of K, you should:

1. Read `p_weight_scale` as a per-output-channel vector indexed by the output
   column `n` (i.e. `tile_n * BLOCK_TILE_SIZE_N + n_in_tile`), **not** with the
   block-wise `n_block * (K // 128) + k_block` indexing.
2. Stop reloading / prefetching the weight scale inside the K loop. There is one
   scale per output channel for the whole K reduction.
3. Accumulate the FP8 matmul over the full K first, then multiply the final
   accumulator by `W_scale[n]` once (broadcast over M), right before writing the
   output — instead of multiplying a per-K-block scale into each partial result.

The dequant becomes:

```
acc_fp32[m, n] = sum_k  A[m, k] * W_fp8[n, k]        # over the full K
out[m, n]      = acc_fp32[m, n] * W_scale[n]
```

Keep everything else (input load, weight shuffle handling, MFMA layout, the
bf16 branch, the `num_warps == 1` and multi-warp output paths) working as-is.

## Deliverable

Edit `/app/gemm_splitk.py` in place. The graded entrypoint is the Gluon kernel
`gemm_splitk_kernel(p_input, p_weight, p_output, p_weight_scale, M,
weight_dtype, K, N, BLOCK_TILE_SIZE_M, BLOCK_TILE_SIZE_N, SHUFFLE=0,
num_warps=4)`. The verifier calls it exactly as the file's own helpers do
(`shuffle_weight`, `div_up`, `SHUFFLE` must remain importable from the module),
passing a PTPC weight scale of shape `[N, 1]`.

## How you are graded

1. **Correctness** — on FRESH random PTPC inputs across several

   `num_warps=4`), your output must match the torch reference

repo is at `/app/repo`.
